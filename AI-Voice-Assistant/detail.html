一、硬件端详细技术实现

1.1 主控单元：STM32F103RCT6

  • 内核：ARM Cortex-M3，最高72MHz工作频率
  • 存储：256KB Flash，48KB SRAM
  • 外设资源：3个USART、2个I2C、2个SPI、2个I2S、12位ADC×2（16通道）、51个GPIO
  • 封装：LQFP-64

1.2 离线语音识别模组：SU-03T

参数数值
核心处理器32bit RISC，240MHz，支持DSP+FPU
语音识别算法神经网络唤醒引擎 + 语音识别引擎
支持命令词数量最多50条（可自定义）
识别响应时间<300ms
工作模式串口输出识别结果（ID或预设字符串）
供电5V，平均电流<80mA

配置方式：通过USB连接PC，使用官方配置工具（SU-03T Config Tool）烧录命令词表。每条命令词可绑定一个4字节的识别码，STM32通过串口接收该识别码，执行对应的动作。

信号链路：驻极体麦克风（模拟信号） → SU-03T内部ADC → 语音特征提取 → 神经网络推理 → 识别码 → UART TX（TTL电平，波特率115200） → STM32 USART1接收。

1.3 温湿度传感器：AHT25

  • 通信接口：I2C（标准模式100kHz，快速模式400kHz）
  • 测量范围：温度-40~+80℃，湿度0~100%RH
  • 精度：典型值±0.3℃（温度），±2%RH（湿度）
  • 分辨率：0.01℃ / 0.024%RH
  • 采集周期：建议100ms一次，实际项目中设置为每2秒采集一次，降低总线负载。
  • I2C地址：0x38（7位）

读取流程：发送初始化命令（0xBE）→等待80ms→发送测量触发命令（0xAC+0x33+0x00）→等待80ms→读取6字节数据（温湿度原始值）→CRC校验→转换为实际物理量。

1.4 液晶显示屏：1.47寸TFT

  • 驱动IC：ST7789V（SPI接口）
  • 分辨率：172×320（RGB 65K色）
  • 接口引脚：SCL、SDA、DC、RST、CS、BL（背光控制）
  • 显示内容：当前对话状态（录音中/识别中/回复中）、温湿度数值、网络连接状态、时间（需外置RTC或由服务器授时）。

显示刷新策略：采用双缓冲机制，使用STM32的DMA+SPI传输，帧率可达30fps以上，保证UI流畅。

1.5 音频采集与播放进阶设计（针对自由对话需求）

BOM V1.1中SU-03T不支持输出原始音频，为实现云端大模型自由对话，额外增加以下硬件：

a) 数字麦克风：INMP441（I2S接口）

参数数值
信噪比（SNR）61dBA
灵敏度-26dBFS
工作电压1.8~3.6V
输出格式I2S，24位，采样率可配置（8kHz~96kHz）
连接方式SCK（BCLK）、WS（LRCK）、SD（DATA）

STM32 I2S配置：

  • 工作模式：主模式接收（Master RX）
  • 数据格式：16位或24位左对齐，PCMSYNC模式
  • 采样率：16kHz（兼顾语音质量和网络带宽）
  • DMA双缓冲：使用两个缓冲区乒乓接收，避免数据丢失。

音频预处理流水线（在STM32内完成）：

  1. 从I2S DMA缓冲区读取原始PCM（16kHz，16bit）
  2. 预加重滤波：一阶高通，系数0.97
  3. 分帧：25ms帧长，10ms帧移（每帧400采样点）
  4. 加汉明窗：减少频谱泄漏
  5. VAD检测：计算每帧能量（对数域），当连续3帧超过阈值时判定为语音开始，连续10帧低于阈值时判定为结束
  6. 降噪（可选）：谱减法或MMSE，仅在低信噪比环境下启用（通过能量方差判断环境噪声等级）

b) 音频功放与喇叭

  • 功放芯片：NS8002（AB类，3W输出，SOP-8封装）
  • 输入信号：STM32的DAC输出（12位分辨率）或PWM滤波输出
  • 连接方式：音频信号经10uF隔直电容 → 功放IN+ → 功放OUT+/OUT- → 喇叭（4Ω，3W）

音频播放流程：服务器返回的OPUS音频 → STM32解码（或透传给硬件解码器） → 转换为PCM → 通过DMA传输到DAC → 功放 → 喇叭发声。

1.6 无线通信模块：ESP-01S (ESP8266)

  • 核心：ESP8266，80MHz，内置TCP/IP协议栈
  • 工作模式：STA模式，连接家庭/实验室WiFi
  • 接口：UART2与STM32连接（TX/RX，波特率115200）
  • 通信协议：AT指令集实现WebSocket连接

AT指令序列示例：

AT+CIPSTART="WS","ws://服务器IP:8000/ws/voice",,,"音频数据"

STM32通过轮询或空闲中断接收服务器返回的音频数据（二进制帧），实时送入扬声器播放。

为保证实时性，采用以下优化：

  • 将ESP-01S的串口接收缓冲区设为2KB，DMA接收
  • 服务器端采用分块传输，每块音频长度约200ms，降低端到端延迟
  • 心跳包：每30秒发送一次ping，保持WebSocket长连接

1.7 电源管理

  • 输入：Mini USB (5V/2A)
  • 稳压：AMS1117-3.3，输出3.3V给STM32、传感器、液晶、SU-03T（部分）供电
  • 保护：自恢复保险F1（1812封装，1.1A），防反接二极管D3（1N5819WS）
  • 去耦电容布局：每个电源引脚就近放置0.1uF陶瓷电容，大容量钽电容（47uF/16V）用于稳定瞬态电流

1.8 硬件端状态机与任务调度

采用FreeRTOS进行任务划分：

任务名称优先级周期/触发功能
AudioCaptureTask高I2S DMA半满中断读取麦克风PCM，存入环形缓冲区
VADTask高每当新帧到达检测语音活动，设置语音标志位
AudioSendTask中当语音段结束时压缩PCM为OPUS，通过队列发送给WiFi任务
WiFiRecvTask中串口空闲中断接收服务器音频数据，存入播放缓冲区
AudioPlayTask中DMA传输完成中断从播放缓冲区取数据送DAC
UIDisplayTask低200ms更新LCD显示（温度、状态）
SensorReadTask低2s读取AHT25温湿度
KeyScanTask低20ms扫描按键，处理用户输入

二、AI服务器端详细技术实现

2.1 整体架构

  • Web框架：FastAPI（Python，异步支持）
  • WebSocket端点：/ws/voice，处理硬件端、网页端、手机端的音频流
  • HTTP端点：/api/chat（文本对话）、/api/health（健康检查）

2.2 语音识别（ASR）

采用Whisper（OpenAI开源模型）的本地部署版本：

参数选择
模型版本base（74M参数）或small（244M参数）
语言中文（--language zh）
推理后端faster-whisper（CTranslate2）或原版PyTorch
解码方式集束搜索（beam_size=5）
实时因子0.3~0.5（依赖GPU）

音频预处理：服务器接收到OPUS帧后，先解码为16kHz 16bit PCM，使用libsox进行静音切除（silence效果器），然后送入Whisper模型。

若网络条件允许，也可调用阿里云智能语音交互或讯飞语音听写的API，利用其商业级准确率和多方言支持。

2.3 大模型集成（双方案）

方案A：阿里云百炼 + GPT-OSS-20B（推荐）

  • 模型名称：gpt-oss-20b
  • API端点：https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation
  • 认证方式：API Key（从阿里云控制台获取）
  • 计费：新用户赠送9000万Tokens免费额度，足够大创项目全程使用
  • 部署优势：无需自备GPU，即开即用，支持高并发流式输出

调用参数：

{ "model": "gpt-oss-20b", "input": { "messages": [ {"role": "system", "content": "你是一个智能语音助手..."}, {"role": "user", "content": "用户文本"} ] }, "parameters": { "max_tokens": 1024, "temperature": 0.7, "top_p": 0.9, "stream": true } }

方案B：本地Ollama + GPT-OSS-20B（离线部署）

  • 部署方式：ollama run gpt-oss-20b
  • 硬件要求：NVIDIA RTX 4090（24GB VRAM）或同等级别
  • 量化方法：使用MXFP4量化，显存占用降至16GB，普通消费级显卡可运行
  • 推理速度：10~15 tokens/s（端到端对话延迟1~3秒）

2.4 Agent技能系统：OpenClaw深入集成

OpenClaw（龙虾）是一个开源的AI Agent执行网关，支持私有化部署，数据本地可控。

架构层级：

  • 控制网关层：负责请求路由、API签名、安全沙箱（限制敏感操作）。
  • 智能体单元：基于ReAct模式（Reason + Act），循环执行“思考→行动→观察”直到任务完成。
  • 技能扩展层：所有技能以Python函数或HTTP API形式注册，Agent根据用户意图自动选择调用。

与LLM的交互流程：

用户文本 → Agent接收 → 构造system prompt（包含可用工具列表） → 调用LLM → LLM返回function call（工具名称+参数） → Agent执行函数 → 获取结果 → 再次调用LLM（包含函数调用结果） → 生成最终答案

技能来源：

  • 自主培养：编写自定义Python工具（如控制实验室灯、查询课程表），注册到OpenClaw
  • 开源Hub：从AgentSkills.cc下载社区技能，一键导入
  • API集成：使用Composio等插件，将GitHub、Gmail、飞书等第三方服务作为技能

2.5 语音合成（TTS）

方案特点使用场景
Piper（本地）离线，轻量，支持40+语言，音色可定制完全离线环境
Edge TTS API免费，音质好，在线临时测试或演示
阿里云TTS商业级，支持情感和语速调节需要高质量语音时（消耗免费额度）

本项目采用Piper的zh_CN-huayan-medium模型，输出16kHz PCM，再用OPUS编码回传，保证低带宽消耗。

2.6 音频编解码与传输优化

  • 编码器：libopus（C库），在服务器和硬件端均使用
  • 编码参数：bitrate=32kbps，complexity=5（平衡CPU与压缩比），frame_size=20ms
  • 传输协议：WebSocket（二进制帧）
  • 打包策略：服务器每生成200ms音频（10帧OPUS），立即发送一个WebSocket帧，实现流式播放，降低首字延迟。

2.7 云端部署（阿里云ECS）

  • 实例规格：ecs.g6.xlarge（4核16GB，若有GPU可选gn6i）
  • 操作系统：Ubuntu 22.04 LTS
  • 部署工具：Docker + Docker Compose（包含Whisper、OpenClaw、TTS等容器）
  • 反向代理：Nginx（处理WebSocket升级，SSL终止）
  • 监控：阿里云CloudMonitor（CPU、内存、网络流量）

三、软件端（网页与移动端）技术实现

3.1 网页前端

  • 技术栈：原生HTML5 + Tailwind CSS + Vanilla JavaScript
  • 录音实现：MediaRecorder API，音频格式webm（Opus），采样率16kHz
  • WebSocket连接：浏览器原生WebSocket API，二进制帧接收和发送
  • PWA支持：配置manifest.json和Service Worker，实现“添加到主屏幕”功能，离线时显示缓存界面。

代码结构：

let ws = new WebSocket('wss://server/ws/voice'); ws.binaryType = 'arraybuffer'; // 录音 navigator.mediaDevices.getUserMedia({ audio: true }) .then(stream => { mediaRecorder = new MediaRecorder(stream, { mimeType: 'audio/webm' }); mediaRecorder.ondataavailable = (e) => { if (ws.readyState === WebSocket.OPEN) ws.send(e.data); }; });

3.2 移动端打包

由于网页已适配移动端（响应式CSS），使用Capacitor或PWA直接打包为Android/iOS应用，无需重复开发原生。Capacitor会将网页包裹在WebView中，并提供原生插件访问录音权限。

四、整体数据流与协议规范

4.1 上行（客户端→服务器）消息格式

字段类型说明
typestring固定为"audio"或"text"
database64音频数据（OPUS编码）或文本内容
sequint32包序号，用于丢包检测和重传
timestampuint64毫秒级时间戳（用于延迟统计）

服务器收到后，回复ACK（序号确认）。

4.2 下行（服务器→客户端）消息格式

字段类型说明
typestring"audio"/"text"/"status"
database64音频PCM-OPUS或文本回复
seq_ackuint32确认收到的上行包序号

当type="status"时，data包含JSON状态信息（如{"state":"processing"}）。

4.3 端到端延迟拆解

阶段典型耗时
硬件端采集+OPUS编码40ms
WiFi上传10~30ms
服务器ASR200~400ms
LLM推理（云端）500~1500ms
TTS合成100~200ms
OPUS编码+下行30ms
硬件端解码播放10ms
总计约900~2200ms

通过优化（模型量化、流式输出）可降低至1秒以内。

五、自主培养Skills的完整技术链路

目标：训练专门针对智能家居控制或实验室设备的技能，提高特定指令的准确率。

数据采集

使用阿里云PAI DataWorks + 语音标注模板，录制并标注2000条命令（如“打开空调”、“设定温度26度”）。

离线增强：对原始音频添加噪声、变速、音量变化，扩充至8000条。

预处理

音频→Whisper转录为文本→过滤识别错误→人工校对→生成（文本，意图，槽位）三元组。

训练流程

选择预训练模型（如Qwen-2B或ChatGLM3-6B），使用LoRA方法微调，训练目标为function calling准确率。

训练环境：阿里云PAI-DSW（单卡V100），batch_size=8，学习率2e-4，训练5个epoch。

部署

微调后的LoRA适配器与基础模型合并，上传至Ollama或vLLM。

在OpenClaw中配置新的function schema，并链接到合并后的模型。

验证

在测试集上计算意图识别F1-score，目标>90%。
